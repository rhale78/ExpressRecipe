# HighSpeedDAL Framework Usage Tests

## Purpose

These tests demonstrate **real-world framework usage** with source-generated DAL classes. Unlike isolated component tests in other test projects, these validate the **full framework stack**:

```
Entity attributes ? Source generator ? Generated DAL ? Database
```

## For Developers

**USE THESE TESTS AS EXAMPLES** for your own projects. The entity definitions and DAL usage patterns shown here are **exactly how you should structure your code**.

## Why This Project Exists

### The Problem

Other test projects (Core.Tests, SqlServer.Tests, Sqlite.Tests) use manual test entities without `[Table]` attributes and implement manual cloning via `IEntityCloneable<T>`. They test **isolated components** (good for unit testing) but **don't demonstrate framework usage** (bad for learning).

**Example of component test** (isolated testing):
```csharp
// Manual entity - NO source generation
public class SqlServerTestProduct : IEntityCloneable<SqlServerTestProduct>
{
    public int Id { get; set; }
    public string Name { get; set; } = string.Empty;
    
    // Manual cloning - framework generates this automatically
    public SqlServerTestProduct ShallowClone()
    {
        return new SqlServerTestProduct { Id = this.Id, Name = this.Name };
    }
}

// Raw SQL - not how users should write code
DataTable dataTable = new DataTable();
using (SqlBulkCopy bulkCopy = new SqlBulkCopy(connection))
{
    await bulkCopy.WriteToServerAsync(dataTable);
}
```

### The Solution (This Project)

**Framework usage tests** demonstrate real-world usage:
```csharp
// Framework entity - source generation enabled
[Table("Products")]
[Cache(CacheStrategy.Memory, ExpirationSeconds = 300)]
public partial class Product  // ? partial enables source generator
{
    // Id property auto-generated by framework
    public string Name { get; set; } = string.Empty;
    public decimal Price { get; set; }
}

// Generated DAL usage (ProductDal created by source generator)
await productDal.BulkInsertAsync(products);  // ? Framework handles SqlBulkCopy
```

## Why Two Types of Tests?

| Test Type | Project | Purpose | Entities | Methods | Source Generator |
|-----------|---------|---------|----------|---------|------------------|
| **Component Tests** | Core.Tests, SqlServer.Tests, Sqlite.Tests | Validate isolated components | Manual, no `[Table]` | Direct component APIs | ? Not used |
| **Framework Usage Tests** | **THIS PROJECT** | Demonstrate real-world usage | `[Table]`, `partial` | Generated DAL methods | ? Used |

**Both are important:**
- **Component tests** ensure individual pieces work correctly (fast, isolated)
- **Framework usage tests** ensure the full stack works correctly (realistic, copy-paste examples)

## Key Differences from Component Tests

### Component Tests (Other Projects)

**Purpose**: Test individual components in isolation
- ? Fast execution (no source generator)
- ? Focused on specific component behavior
- ? Use mocks and manual entities
- ? Don't demonstrate framework usage
- ? Can't be copied to real projects

**Example**:
```csharp
// Manual entity for isolated testing
public class SqlServerTestProduct : IEntityCloneable<SqlServerTestProduct>
{
    public int Id { get; set; }
    public SqlServerTestProduct ShallowClone() { /* manual code */ }
}
```

### Framework Usage Tests (This Project)

**Purpose**: Demonstrate real-world framework usage
- ? Tests full framework stack
- ? Uses source-generated DAL classes
- ? Copy-paste ready for developers
- ? Validates end-to-end workflow
- ?? Slightly slower (source generator runs)

**Example**:
```csharp
// Framework entity for real usage
[Table("Products")]
[Cache(CacheStrategy.Memory)]
public partial class Product { }  // ? Id auto-generated, ProductDal generated
```

### 3. Use Generated DAL Classes

```csharp
// ProductDal is automatically generated by source generator
private readonly ProductDal _productDal;

public async Task DoWork()
{
    var product = await _productDal.InsertAsync(new Product { Name = "Test" });
    var retrieved = await _productDal.GetByIdAsync(product.Id);
    await _productDal.UpdateAsync(retrieved);
}
```

### 4. Configure Dependency Injection

```csharp
services.AddSingleton<IDbConnectionFactory, SqlServerConnectionFactory>();
services.AddSingleton<ProductDal>();
```

## Common Questions

### Q: Why use SQLite for testing instead of SQL Server?

**A**: Fast, in-memory, no external dependencies. Framework works the same with both.

### Q: Can I copy test code to my production project?

**A**: **YES!** That's the point. Entity definitions and DAL usage patterns are production-ready.

### Q: Why don't other tests use source generators?

**A**: Component tests isolate behavior for unit testing. Framework usage tests demonstrate real usage.

### Q: Which tests should I study to learn the framework?

**A**: **This project** (FrameworkUsage.Tests) and `examples/SimpleCrudExample`.

### Q: Do I need to write manual cloning code like in other tests?

**A**: **NO!** Framework handles cloning automatically. Only component tests use manual cloning.

### Q: Should I use `[Table]` attribute in my entities?

**A**: **YES!** (optional but recommended) - enables source generator to create DAL classes.

### Q: Do I need `partial` keyword on my entity classes?

**A**: **YES** if using `[AutoAudit]`, `[SoftDelete]`, or auto-generated Id. Framework generates additional properties.
