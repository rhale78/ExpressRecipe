# HighSpeedDAL Repository Replacement Strategy

## Current Status

The ProductService has been updated to use HighSpeedDAL through adapter classes that bridge the existing repository pattern with the new framework.

## What Was Changed

### 1. Added Adapter Classes

**ProductRepositoryAdapter.cs** - Implements `IProductRepository` using `ProductEntityDal`  
**IngredientRepositoryAdapter.cs** - Implements `IIngredientRepository` using `IngredientEntityDal`

These adapters allow the existing service layer (BatchProductProcessor, OpenFoodFactsImportService, etc.) to continue working without changes while we transition to HighSpeedDAL.

### 2. Updated Program.cs Registration

```csharp
// HighSpeedDAL components
builder.Services.AddSingleton<ProductDatabaseConnection>();
builder.Services.AddSingleton<ProductEntityDal>();  // Auto-generated by source generator
builder.Services.AddSingleton<IngredientEntityDal>();  // Auto-generated by source generator

// Use adapters (new pattern)
builder.Services.AddScoped<IProductRepository, ProductRepositoryAdapter>();
builder.Services.AddScoped<IIngredientRepository, IngredientRepositoryAdapter>();
```

The old `ProductRepository` and `IngredientRepository` classes remain in the codebase but are no longer registered by default. They can be easily re-enabled by uncommenting the fallback registration code in Program.cs.

## Current Limitations

### Critical Issue: Guid vs. Int ID Type Mismatch

**Problem:** The existing codebase uses `Guid` IDs everywhere, but HighSpeedDAL entities use `int` IDs (auto-increment identity columns).

**Impact:** Most adapter methods currently throw `NotImplementedException` because they cannot convert between Guid and int IDs without a mapping strategy.

**Affected Operations:**
- GetByIdAsync(Guid id)
- CreateAsync() - returns Guid
- UpdateAsync(Guid id, ...)
- DeleteAsync(Guid id, ...)
- All product/ingredient relationship operations

### Missing Custom Query Methods

HighSpeedDAL auto-generates basic CRUD operations (GetById, GetAll, Insert, Update, Delete, BulkInsert, etc.) but does NOT auto-generate custom queries like:

- GetByBarcodeAsync
- SearchAsync (with complex filters)
- GetIngredientIdsByNamesAsync
- GetProductByExternalIdAsync
- And many others

These require **custom DAL method implementations** beyond what the source generator provides.

## Next Steps

### Option 1: Full Migration (RECOMMENDED)

Migrate the entire application from Guid IDs to int IDs:

1. **Database Schema Changes**
   - Change all ID columns from UNIQUEIDENTIFIER to INT IDENTITY
   - Update foreign keys
   - Create migration scripts

2. **DTO Updates**
   - Update all DTOs in `ExpressRecipe.Shared` to use `int` instead of `Guid`
   - CreateProductRequest, ProductDto, IngredientDto, etc.

3. **Service Layer Updates**
   - Update all services to work with int IDs
   - Update API controllers

4. **Implement Adapter Methods**
   - Replace NotImplementedException with actual implementations
   - Map between Entity types and DTOs

5. **Add Custom DAL Methods**
   - Extend ProductEntityDal with custom queries (GetByBarcode, Search, etc.)
   - Extend IngredientEntityDal with custom queries

**Timeline:** 2-3 days of focused work  
**Risk:** Medium - requires coordinated changes across many files  
**Benefit:** Full HighSpeedDAL integration with auto-generated code and high performance

### Option 2: Dual ID System (INTERIM)

Keep Guid IDs in the API/service layer but use int IDs in the database:

1. **Add Mapping Table**
   ```sql
   CREATE TABLE ProductIdMapping (
       GuidId UNIQUEIDENTIFIER PRIMARY KEY,
       IntId INT NOT NULL UNIQUE
   )
   ```

2. **Implement ID Translation in Adapters**
   - LookupInt(Guid) and LookupGuid(int) methods
   - Translate at the adapter boundary

3. **Populate Mapping Table**
   - Migration script to create mappings for existing data

**Timeline:** 1-2 days  
**Risk:** Low - minimal changes to existing code  
**Benefit:** Allows gradual migration, maintains API compatibility  
**Drawback:** Performance overhead from ID lookups, increased complexity

### Option 3: Revert to Old Repositories (FALLBACK)

If HighSpeedDAL integration proves too complex:

1. **Uncomment Fallback Registration in Program.cs**
   ```csharp
   // Option 2: Keep old repositories (FALLBACK)
   builder.Services.AddScoped<IProductRepository>(sp => 
       new ProductRepository(...));
   ```

2. **Remove Adapter Classes**

3. **Continue with Manual SQL Repositories**

**Timeline:** 5 minutes  
**Risk:** None  
**Benefit:** Back to working state immediately  
**Drawback:** Lose all HighSpeedDAL benefits (auto-generation, InMemoryTable, bulk ops, etc.)

## Recommendation

**Go with Option 1 (Full Migration)** because:
1. Clean architecture - no translation layer overhead
2. Full HighSpeedDAL feature access (InMemoryTable, bulk ops, auto-generation)
3. Better performance (int IDs are smaller, faster)
4. Long-term maintainability (less code to maintain)

The ID type change is a one-time migration effort that pays dividends in code quality and performance going forward.

## Testing Strategy
